location 块

  • 意义： location指令的作用是根据用户请求的 URI 来执行不同的应用，也就是根据用户请求的网站 URI 进行匹配，匹配成功即进行相关的操作。
  • 语法：location [ = | ~ | * | ^ ] URI { …. }
  • 注释：| 指令 || 匹配标识 || 匹配的网站网址 || 匹配 URI 之后要执行的配置段 |

语法详解 之 语义

语法 语法详解
= 开头表示精确匹配
^~ 以xx开头：开头表示uri以某个常规字符串开头，理解为匹配 url路径即可。nginx不对url做编码，因此请求为/static/20%/aa，可以被规则^~ /static/ /aa匹配到（注意是空格）。
~ 以xx结尾：开头表示区分大小写的正则匹配
~* 以xx结尾：开头表示不区分大小写的正则匹配
!~ 区分大小写不匹配的正则
!~* 不区分大小写不匹配的正则
/ 通用匹配，任何请求都会匹配到。

语法详解 之 优先级

优先级：（从上及下， 依次降低）

  • 精确匹配 =
  • 其次以xx开头匹配^~
  • ~ 区分大小写的正则匹配
  • ~* 不区分大小写的正则匹配
  • 然后是按文件中顺序的正则匹配
  • 最后是交给 / 通用匹配。

当有匹配成功时候，停止匹配，按当前匹配规则处理请求。

location 块内其他元素

alias 与 root

root和alias都可以定义在location模块中，都是用来指定请求资源的真实路径。

root

请求 http://www.abc.com/i/top.gif 这个地址时，在服务器里面对应的真正的资源是 /data/w3/i/top.gif文件

注意：

  • 真实的路径是(root指定的值加上location指定的值)。
alias

请求 http://www.abc.com/i/top.gif 这个地址时，在服务器里面对应的真正的资源是 /data/w3/top.gif

注意：

  • 而 alias 正如其名，alias指定的路径是location的别名，不管location的值怎么写，资源的真实路径都是 alias 指定的路径
  • alias 后面必须要用 “/” 结束，否则会找不到文件，而 root 则对 ”/” 可有可无。

proxy_pass (代理转发)

  • 意义：所谓 代理转发 即将访问的地址 代理转发到其他 ip 地址上去 获取所需要的资源。
  • 用法：设置被代理的服务器的协议和地址，还可以设置可选的URI。协议是“http”或者“https”。
  • 注意：在nginx中配置proxy_pass代理转发时，如果在proxy_pass后面的url加/，表示绝对根路径；如果没有/，表示相对路径，把匹配的路径部分也给代理走。

proxy_http_version

syntax proxy_http_version 1.0 | 1.1
default roxy_http_version 1.0
context http, server, location

proxy_http_version 设置成 1.1 是为了 sse 以及 websocket 这类接口的连接， 因为 http version 1.1 默认 接口连接 keepalive (长连接)。

proxy_read_timeout

  • 语法：proxy_read_timeout time
  • 默认值：proxy_read_timeout 60s

如果处理请求所需的时间比较长，比如需要加载一个耗时 1.7s 的内容，proxy_read_timeout 要设置超过这个时间，否则出现 504 状态码。
如果 504 偶现，怀疑是请求在某些情况下请求耗时超过 read 时间限制或者后端部分机器 read 时间设置问题。

总结：

通过请求的的时长判断，设置合理的超时时间，可以优化 nginx 的性能

proxy_set_header（代理设置标题）

用户认证接口： 根据客户端IP和port，进行IP反查和端口范围确认，如符合则用户认证通过。

当前使用的是Nginx负载均衡，从客户端到Nginx端 ip和port都对，从Nginx到应有服务器上-port端口变成很奇怪的端口号。

疑问：Nginx往应有服务器上 是如何 传递 客户端IP和port 参数的呢？（proxy_set_header 的作用正在于此）

  • proxy_set_header 概念: 允许重新定义或添加字段传递给代理服务器的请求头。该值可以包含文本、变量和它们的组合。

    在没有定义proxy_set_header时会继承之前定义的值。
